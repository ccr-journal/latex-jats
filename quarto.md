# Quarto → JATS comparison (CCR2026.2.11.URMA)

Comparison of `examples/CCR2026.2.11.URMA/CCR2026.2.11.URMA.xml` (produced by `quarto render` with `format: jats_publishing`) against the professional gold XML in `CCR2025.1.2.YAO_gold/CCR2025.1.2.YAO.xml` and our own LaTeX→JATS output in `output/CCR2025.1.2.YAO/convert/CCR2025.1.2.YAO.xml`.

## 1. Validation

Schema used: `src/schema/jats-publishing-1.2-rng/JATS-journalpublishing1-mathml3.rng` (Jing RelaxNG).

- **Our LaTeX→JATS output** (`CCR2025.1.2.YAO.xml`): validates cleanly, **0 errors**.
- **Gold** (`CCR2025.1.2.YAO_gold`): uses a different DTD reference (`JATS-journalpublishing1-mathml3.dtd` by name, not our local copy); structurally clean.
- **Quarto output** (`CCR2026.2.11.URMA.xml`): **3 validation errors**:
  1. `article-meta` not allowed yet; missing required `journal-meta` (line 6). Quarto emits no `<journal-meta>` at all.
  2. `xref` not allowed inside `<string-name>` (line 59). Quarto puts a corresponding-author footnote xref directly inside a `<name>/<string-name>`, which JATS forbids.
  3. `history` incomplete; missing required `date` (line 132). Quarto emits an empty `<history/>`.

DTD validation via `xmllint` was not attempted against the NLM DTDs (network fetch blocked); Jing RNG is authoritative here.

## 2. Front-matter differences

| Element | Gold / our output | Quarto |
|---|---|---|
| DOCTYPE | `JATS-journalpublishing1-mathml3.dtd` (full path/URL) | `JATS-publishing1.dtd` (bare filename) |
| `xml:lang` | `en` on `<article>` | absent |
| `article-type` | not set | `"other"` |
| `<journal-meta>` | full (journal-id, journal-title, issn, publisher) | **missing entirely** |
| `<article-id>` (publisher-id + doi) | present | **missing** (DOI is in the .qmd front matter but not emitted) |
| `<article-categories>` / subject | `Article` | missing |
| `<pub-date>`, `<volume>`, `<issue>`, `<fpage>`, `<lpage>` | present | **missing** (quarto has them in YAML but drops them) |
| `<permissions>` / license / copyright | present | missing |
| `<abstract>` | with `<title>Abstract</title>` | no title; otherwise fine |
| `<kwd-group>` | one `<kwd>` per keyword, `kwd-group-type` unset | `kwd-group-type="author"`, **all keywords in a single `<kwd>` element** (`<kwd>Bias, Gender, Multilingual, LLM, generative AI</kwd>`) |
| `<history/>` | omitted or populated | empty `<history/>` (invalid) |
| affiliations | inline `<aff>` per author (our pipeline) or list with ids (gold uses inline) | id-based `<aff id="aff-N">` referenced via `<contrib>/<xref ref-type="aff">`; wraps in `<institution-wrap>/<institution>` |
| author corresponding-author | uses `corresp` / `author-notes` | Quarto encodes it as an inline `<xref ref-type="fn">` **inside `<string-name>`** → invalid |
| `<string-name>` | our pipeline does not emit it | Quarto emits both `<name>` and `<string-name>` for every author (redundant but allowed, except see above) |

## 3. Body differences

- **Sectioning**: Quarto produces nicely nested `<sec id="...">` with human-readable slugs and proper `<title>` elements. Our pipeline uses autogenerated IDs like `S1`, `Sx1`, `S1.p1`. Both are valid; Quarto is more readable.
- **Paragraphs**: Quarto preserves paragraph content cleanly with `<italic>`, `<bold>`. No odd `<p id="...">` numbering.
- **Citations**: Quarto wraps each citation in `<xref ref-type="bibr" rid="ref-KEY" alt="...">...</xref>`. Gold uses `<xref ref-type="bibr" rid="CIT0001">Author, Year</xref>`. Our pipeline uses `<xref rid="bib.bibx1" ref-type="bibr">Year</xref>`. All three are valid; Quarto additionally includes the `alt` attribute (harmless, JATS-allowed).
- **Figures**:
  - Quarto emits `<fig id="fig-1">` with caption paragraph prefixed by the literal string "Figure 1: ..." — there is **no `<label>` element**. Gold and our output both emit `<label>Figure 1:</label>` separately from the `<caption>/<title>`.
  - Figure graphics: Quarto references `figures/svg/figure_1.svg`; our pipeline already converts PDFs to SVG and renames to `ID_figN.ext`. The Quarto output would need the same file-rename and copying step.
- **Tables**:
  - Quarto wraps tables inside `<fig id="tbl-2">` (i.e. a `<fig>` containing a `<table-wrap>` containing the actual `<table>`). This is **not how gold/our output represents tables** — normal JATS uses `<table-wrap id="T1">` at the block level, not nested in a `<fig>`. The nested `<fig>` wrapper is a Quarto idiosyncrasy and should be unwrapped.
  - When a table has no body (e.g. tbl-2, tbl-3) Quarto emits an empty `<fig>` with only a caption — they are placeholders for tables that were meant to be rendered from R/knitr chunks but weren't. These will show as missing tables in the final PDF/HTML.
  - No `<label>` element; label ("Table 2:") is inlined into the caption text.
  - Tables use `<th>`/`<td>` cleanly with `<colgroup>`; that matches our output.
- **Lists**: `<list list-type="bullet">` with nested `<list-item><p>...</p></list-item>`. Matches our output.
- **Math**: Quarto uses `<inline-formula><alternatives><tex-math>...</tex-math><mml:math>...</mml:math></alternatives></inline-formula>`. Our pipeline emits only `<mml:math>` wrapped in `<inline-formula>`. Quarto's `alternatives` form is richer and preferred.
- **Footnotes**: Quarto places all footnotes in a `<fn-group>` **inside `<body>` at the very end** (lines 5449–5564). Gold/our output put `<fn-group>` inside `<back>`. Quarto's location is actually still valid (JATS allows fn-group in body), but we normalize to `<back>` via `fix_footnotes`.
- **References**:
  - Quarto puts references into `<back><ref-list><ref id="ref-KEY"><element-citation>` with highly structured fields (person-group, article-title, source, year, volume, issue, fpage, lpage, doi, uri). Our pipeline emits `<mixed-citation>` (flat marked-up text). **Quarto's element-citation is the richer, more canonical JATS form.** Gold uses a simpler structured form (person-group + name + source + year).
  - Quarto additionally leaves an **empty `<sec id="references"><title>References</title><p/></sec>` in the body** alongside the real ref-list in `<back>`. This empty section should be stripped.
  - Quarto emits `<date-in-citation content-type="access-date">` for every reference (even non-web ones), populated with today's date. This is noise and should probably be removed or only kept for online sources.
- **Appendix**: Quarto places the appendix as a `<sec id="appendix">` inside `<body>`. Our pipeline moves appendices into `<back><app-group>`. Both are valid.

## 4. Tag/structure inventory summary

Missing from Quarto vs. what gold + our pipeline produce:

- `<journal-meta>` (blocking — causes invalid XML)
- `<article-id>` (publisher-id + doi)
- `<article-categories>` / `<subj-group>`
- `<pub-date>`, `<volume>`, `<issue>`, `<fpage>`, `<lpage>`
- `<permissions>` / `<license>` / `<copyright-*>`
- `<label>` on figures and tables
- Proper `<table-wrap>` at block level (instead of `<fig>`-wrapped)
- `<history>` should be populated or omitted

Differently structured (not wrong, but needs harmonization):

- `<fn-group>` in `<body>` vs `<back>`
- `<element-citation>` (Quarto) vs `<mixed-citation>` (ours) — prefer element-citation
- Appendix in `<body>` vs `<back><app-group>`
- Figure/table IDs (`fig-1`, `tbl-2` vs `S1.F1`, `T1`)
- Math uses `<alternatives>` (Quarto) vs bare mml (ours)

Extraneous / invalid in Quarto:

- `<xref>` inside `<string-name>` (invalid)
- Empty `<history/>` (invalid)
- Redundant `<string-name>` alongside `<name>`
- Empty placeholder `<fig id="tbl-2">` etc. with no table body
- Empty `<sec id="references">` in body plus real `<ref-list>` in back
- `<date-in-citation>` access-date on every reference

## 5. Harmonization strategy

The cleanest path is **a Quarto-specific post-processing pipeline in `src/latex_jats/` (e.g. `quarto.py`) that reuses the existing fixup functions and adds a few new ones.** The Quarto XML is much closer to final JATS than raw LaTeXML output, so we need *fewer* fixups, not more.

### Conversion pipeline for quarto input

1. **Run `quarto render --to jats`** (or the existing `format: jats_publishing`) on the `.qmd` file to produce the initial XML.
2. **Apply post-processing** in order:
   - `inject_journal_meta` — build `<journal-meta>` from CCR constants (reuse logic from `fix_metadata`).
   - `inject_article_meta` — read YAML front matter (`doi`, `volume`, `pubnumber` → issue, `pubyear`, `firstpage`, etc.) and inject `<article-id>`, `<article-categories>`, `<pub-date>`, `<volume>`, `<issue>`, `<fpage>`, `<permissions>`. The `.qmd` YAML already contains all the needed fields — parse it directly from the source file (simpler than grubbing around in the XML).
   - `fix_history` — drop empty `<history/>`.
   - `fix_corresp_xref` — move `<xref ref-type="fn">` out of `<string-name>` into a proper `<author-notes>` / `<corresp>` block.
   - `unwrap_table_fig` — unwrap `<fig id="tbl-*">` containing a `<table-wrap>` into a bare `<table-wrap id="tbl-*">`; warn on empty table placeholders (tbl-2, tbl-3 in this example) since they indicate missing content.
   - `add_fig_labels` — parse caption text for leading "Figure N:" / "Table N:" and lift into `<label>`.
   - `move_fn_group` — move `<fn-group>` from end of `<body>` into `<back>` (reuse or adapt `fix_footnotes`).
   - `drop_empty_refs_sec` — remove empty `<sec id="references">` from body when `<back><ref-list>` exists.
   - `clean_citations` — drop noisy `<date-in-citation>` access-dates on non-web references; optionally normalize `<element-citation>` attributes.
   - `move_appendix_to_back` — move `<sec id="appendix">` from body to `<back><app-group>`.
   - `fix_pdf_graphic_refs`, `fix_ext_links`, `finalize_xml` — reuse existing fixups.
3. **Figure files**: copy `figures/svg/*.svg` into the output directory and rename to `<doi-suffix>_figN.svg` using the existing rename logic.

### Design notes

- Quarto's YAML front matter is the authoritative source of article metadata — parse it once from the `.qmd` file instead of scraping the XML. Much cleaner than the LaTeX preamble path.
- Warn (via `logging.warning`) when `<fig id="tbl-*">` has no table body, so authors know their R/knitr table chunk did not render. This matches the project principle of "warn, don't silently fix."
- Reuse `convert_to_html` and the existing jats-html XSLT unchanged — once the XML is harmonized to the same shape as our LaTeX output, the HTML preview should just work.
- Add integration tests with a minimal `.qmd` fixture under `tests/fixtures/quarto/` mirroring the pattern of `tests/fixtures/latex/`.

### CLI shape

Extend `latex-jats` (or add a `quarto-jats` entry point) that detects `.qmd` input and dispatches to the quarto pipeline instead of `latexmlc`. The post-processing chain can mostly be shared.

## 6. Bottom line

Quarto's JATS output is structurally **much closer to publisher-ready JATS** than raw LaTeXML output is — the body, figures, citations, math, and references are cleaner and more canonical. The bad news is almost entirely concentrated in the front matter (missing journal-meta, article-meta fields, license) plus three small but invalidating bugs (xref inside name, empty history, bare DTD filename).

Harmonizing Quarto should be **significantly less work** than the LaTeX pipeline: roughly half a dozen new post-processing functions plus YAML parsing, and the existing fixups for graphics, ext-links, and finalization carry over directly.
